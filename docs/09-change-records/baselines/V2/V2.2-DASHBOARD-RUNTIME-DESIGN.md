# V2.2 Dashboard Runtime Design

Status: Draft baseline

Version: V2.2-DESIGN

Date: 2026-06-07

## Purpose

Define the first code-bearing implementation target for V2 by replacing the current dashboard placeholder with a complex, operations-style runtime surface.

## Existing Baseline

Current dashboard behavior is a simple heading and explanatory paragraph. It does not yet provide:

- meaningful information density
- multiple runtime states
- nested interactions
- evidence-grade selector targets
- realistic operator workflows

## Design Goal

The new dashboard should act as an operational cockpit that is immediately useful for:

- route-level visual verification
- state-observation experiments
- filter and drawer interaction tests
- UI evidence capture
- future incremental coding and drift-diagnosis exercises

## Proposed Page Structure

### Region 1: Page Header

Should include:

- page title
- short operational summary
- primary actions such as refresh or create-follow-up
- time range or environment selector

### Region 2: Summary Metric Band

Should include several status cards representing different operational buckets, such as:

- active workload
- at-risk items
- blocked items
- recent failures or alerts

These cards should be individually interactive or filter-driving rather than decorative only.

### Region 3: Alert or Exception Queue

Should include:

- prioritized items requiring attention
- severity or status tags
- quick actions
- ability to open richer detail without full-page navigation

### Region 4: Activity Timeline or Event Feed

Should include:

- recent changes
- mixed event types
- visual structure suitable for later screenshot and evidence experiments

### Region 5: Detail Drawer or Expandable Panel

Should support:

- opening contextual detail from a summary card or queue item
- preserving dashboard context while inspecting one item
- structured subcontent such as metadata, status history, or follow-up actions

## Required Runtime States

The dashboard implementation should intentionally support:

- default populated state
- loading or skeleton state
- empty state
- error state
- filtered state after user interaction

## Interaction Design Requirements

- At least one summary card click should alter another region on the page.
- At least one queue item should open a drawer or expandable detail surface.
- The time-range or filter controls should produce a visible state transition.
- Interactions should avoid full page reloads or route changes where a local state transition is enough.

## Testability Requirements

The dashboard should expose:

- stable heading and landmark structure
- role-driven controls where practical
- deterministic empty/loading/error toggles through local mock state
- state labels that are easy to assert in Playwright

## Implementation Constraints

- Keep page logic inside `src/domains/dashboard/`.
- Shared primitives may be introduced only when they are likely reusable by task-list or task-create work.
- Prefer local mock data modules over hidden hardcoded JSX blobs.
- Keep styles and composition aligned with the existing Ant Design baseline unless a deliberate divergence is justified.

## Verification Direction

V2.2 should plan to extend the current smoke baseline with at least:

- dashboard loaded-state verification
- one interaction-driven state transition
- one drawer or detail-region visibility assertion
- one non-happy-path state assertion

## Conclusion

V2.2 should not optimize for visual polish first. Its main job is to establish the first rich, governable, testable runtime surface in the repository.
