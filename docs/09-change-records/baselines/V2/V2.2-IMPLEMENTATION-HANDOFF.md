## Objective

Implement the first rich runtime surface under `src/domains/dashboard/` with enough complexity to validate the V1 control plane against real frontend work.

## Required Outcome

The implemented dashboard must include all of the following:

1. A real page header with title, operational summary, and at least one primary action.
2. A summary metric band with multiple cards.
3. At least one interaction where clicking a summary card changes another region on the page.
4. An alert/exception queue or equivalent operator-facing list.
5. An activity timeline/event feed or similarly structured recent-activity region.
6. A detail drawer or expandable panel opened from the dashboard itself.
7. Explicit runtime states:
   - default populated
   - loading
   - empty
   - error
   - filtered/interacted state
8. Stronger E2E coverage than the current single reachability smoke test.

## Scope Boundary

### Allowed

- `src/domains/dashboard/**`
- `src/shared/**` only when a shared primitive is genuinely justified
- `src/testability/**` if stable selectors or helpers are needed
- `tests/e2e/**`
- `docs/07-evidence/**` if an implementation verification record is added
- `docs/09-change-records/baselines/V2/**` for gate/review/verification records if requested in the same change set

### Forbidden

- `src/domains/task/**` except read-only inspection
- `src/domains/auth/**`
- backend integration of any kind
- unrelated shell redesign
- broad visual redesign outside the dashboard needs
- weakening the dashboard into static cards with no meaningful interactions

## UX and Interaction Contract

### Page Header

Must expose:

- visible heading
- short descriptive summary
- one or more action controls such as refresh, inspect, or create follow-up
- one filter or time-range control

### Metric Band

Must expose:

- several cards representing distinct operational buckets
- visible counts or status summaries
- clear active/inactive interaction state

At least one card must drive a visible filter or focus change elsewhere on the page.

### Queue Region

Must expose:

- operator-facing items with status/severity signal
- at least one action or affordance per item
- one path to open richer context without route navigation

### Activity Region

Must expose:

- mixed event entries or structured recent changes
- enough visible text/structure for future screenshot and evidence experiments

### Drawer or Expandable Detail

Must expose:

- contextual details for one selected item
- visible open/close behavior
- structured subcontent such as metadata, status history, notes, or next actions

## State Model Contract

The dashboard should support deterministic local state switching. Avoid hidden random logic.

At minimum there must be a practical way to render:

- populated state
- loading state
- empty state
- error state
- filtered state

Recommended approach:

- local mock data plus local state
- optional developer-visible segmented control, tabs, or inline trigger to switch scenarios

If scenario switching is exposed in the UI, it should remain useful for testing and evidence, not look like debug debris.

## Testability Contract

Use accessible selectors first. Favor:

- heading roles
- button roles
- region labels
- table/list semantics
- drawer/dialog semantics

`data-testid` is allowed only when accessible selectors are insufficient or unstable.

The page should make it easy to assert:

- dashboard loaded
- metric-card interaction changes queue/feed content
- drawer opens
- empty/error state is visible

## E2E Change Requirements

Update `tests/e2e/` to go beyond current smoke coverage. The new suite should cover at least:

1. dashboard renders its real runtime surface
2. one primary interaction changes visible state
3. one detail drawer/expanded panel opens
4. one non-happy-path state is reachable and assertable

It is acceptable to keep the current smoke and add one or more additional specs, or to expand the existing spec if clarity remains high.

## Verification Commands

Run at minimum:

```powershell
npm.cmd run build
npm.cmd run test:e2e
```

If new lint-sensitive patterns are introduced and lint is stable in the repo, also run:

```powershell
npm.cmd run lint
```

If `lint` is not run, the final report must state that explicitly.

## Implementation Sequence

Execute in this order:

1. Read governing inputs.
2. Inspect current dashboard and existing shell layout.
3. Define local mock data and state model.
4. Implement the page structure and interactions.
5. Add or refine reusable components only if actually justified.
6. Update or add Playwright coverage.
7. Run verification commands.
8. Record outcomes in the final report.

## Review Checklist

Before considering the work complete, confirm:

- placeholder paragraph is gone
- dashboard is not static-only
- at least one cross-region interaction exists
- drawer/expandable detail exists
- loading/empty/error/filtered states exist
- E2E coverage is meaningfully stronger than the old smoke baseline
- no task-list or task-create work was pulled in unnecessarily

## Risks to Avoid

- building a pretty but flat dashboard
- over-engineering shared abstractions before reuse is proven
- hiding all state complexity inside one oversized page component
- adding fake complexity that is impossible to test cleanly
- leaking this iteration into task-list scope

## Acceptance Standard

The work should be rejected if any of the following are true:

- dashboard remains functionally close to a placeholder
- interactions do not materially change visible state
- no explicit non-happy-path state exists
- E2E coverage remains effectively the same as before
- implementation spills into unrelated runtime domains without justification
