# V1.1 Dispatch Contract

Status: Proposed

Version: V1.1-DISPATCH-CONTRACT

Date: 2026-06-07

## 1. Purpose

Define the standard input format, execution constraints, and output requirements for any agent dispatched to perform V1.1 work. This contract ensures that agent tasks are bounded, reviewable, and traceable. It is the authoritative reference for how the design lead delegates work and how the user verifies the result.

## 2. V1.1 Scope Reminder

V1.1 produces governance documents only. No agent dispatched under V1.1 may modify `src/`, `tests/`, `openspec/`, or claim that the governance model is active. V1.1 deliverables are:

- gate contract
- dispatch contract
- gate records (review, disposition, closure for IR, SR, and implementation)
- acceptance precheck and precheck verification records

## 3. Standard Dispatch Input Format

Every agent task must include the following fields in the dispatch message:

### 3.1 Required Fields

| Field | Description | Example |
| --- | --- | --- |
| `target branch` | The git branch the agent must work on | `master` or `v1.1-governance` |
| `target minor version` | The minor version this task belongs to | `V1.1` |
| `target stage` | The gate stage this task contributes to | `IR review`, `SR disposition`, `implementation closure` |
| `allowed files` | Exact file paths the agent may create or modify | `docs/09-change-records/baselines/V1/V1.1-IR-REVIEW.md` |
| `forbidden files` | File paths the agent must not touch under any circumstance | `src/**`, `tests/**`, `openspec/**`, `docs/09-change-records/baselines/V1/IR.md` (unless explicitly refining) |
| `required reads` | Files the agent must read before starting work | `docs/09-change-records/baselines/V1/IR.md`, `docs/09-change-records/current/DECISIONS.md` |
| `validation commands` | Commands the agent must run to verify its output is valid | `npm run lint`, `npm run build` (if any code-adjacent doc changed) |
| `evidence output path` | Where the agent must place its output artifact | `docs/09-change-records/baselines/V1/V1.1-IR-REVIEW.md` |
| `final report format` | The structure the agent must return upon completion | See Section 7 |

### 3.2 Optional Fields

| Field | Description | Example |
| --- | --- | --- |
| `blocking items` | Known blocked items from prior gates the agent must carry forward | "IR review gap #3: missing traceability for IR-V1-007" |
| `user decisions` | Relevant entries from `DECISIONS.md` the agent must respect | "D010: stricter gate model required" |
| `priority` | Urgency classification | `P0` (blocks downstream), `P1` (intra-stage), `P2` (improvement) |

## 4. Agent Prohibitions

An agent dispatched under V1.1 must NOT:

1. Modify any file outside its `allowed files` list.
2. Modify `src/`, `tests/`, or `openspec/` implementation code.
3. Claim a gate has passed. Only the gate records (reviewed by the design lead and accepted by the user) determine gate passage.
4. Declare V1.1 complete or accepted.
5. Start work on V1.2, V1.3, or V1.4 deliverables.
6. Create new baseline documents not listed in the dispatch.
7. Drop or weaken a requirement from IR or SR without an explicit entry in `DECISIONS.md`.
8. Skip any `required reads` file.
9. Produce output that does not follow the `final report format`.
10. Merge its own work into the target branch without design lead review.

## 5. Agent Completion Requirements

When an agent completes its task, it must return:

1. The output artifact at the specified `evidence output path`.
2. A final report (see Section 7) summarizing what was done, what was found, and what remains.
3. Confirmation that all `required reads` were consumed.
4. Confirmation that all `validation commands` were executed (if applicable).
5. A list of any blocked items carried forward from prior gates.
6. A list of any new blocked items discovered during this task.
7. A list of any items that need user confirmation (classified as "needs user confirmation").

## 6. Design Lead Acceptance of Agent Output

The design lead must verify the following before accepting an agent's output:

1. The output artifact exists at the specified path.
2. The artifact content matches the task's `target stage` requirements.
3. No `forbidden files` were modified (check `git diff`).
4. All `required reads` were actually consumed (the agent's report references them or the artifact demonstrates awareness of their content).
5. Blocked items from prior gates are carried forward, not dropped.
6. New blocked items are explicitly listed.
7. Items needing user confirmation are classified correctly.
8. The output follows the required report format.
9. The agent did not exceed its `allowed files` boundary.

If any check fails, the design lead must reject the output and return it to the agent with a specific gap list. The agent must fix the gaps before resubmission.

## 7. Final Report Format

Every agent must return its results in this structure:

```markdown
## Agent Task Report

### Task Identity
- **Dispatch ID**: <unique identifier or description>
- **Target branch**: <branch>
- **Target minor version**: <V1.1>
- **Target stage**: <stage name>
- **Date**: <completion date>

### Work Summary
<1-3 sentences describing what was done>

### Required Reads Consumed
- [ ] <file 1>
- [ ] <file 2>
- [ ] ...

### Output Artifact
- **Path**: <evidence output path>
- **Status**: completed / blocked

### Validation Commands Executed
- [ ] <command 1>: pass / fail / not applicable
- [ ] <command 2>: pass / fail / not applicable

### Blocked Items Carried Forward
| Item | Source Gate | Reason |
| --- | --- | --- |
| <item> | <prior gate> | <reason> |

### New Blocked Items Discovered
| Item | Reason |
| --- | --- |

| Item | Classification | Reason |
| --- | --- | --- |

| Item | Status |
| --- | --- |

<What was not done and why it is out of scope for this task>
```

## 8. User Secondary Review

The user performs the final review after the design lead has accepted the agent output. The user's review checklist:

1. Read the agent's output artifact.
2. Read the agent's final report.
3. Verify that blocked items are genuine blockers, not artificial delays.
4. Verify that "needs user confirmation" items are correctly classified.
5. Verify that no confirmed requirement from IR or SR was weakened without a recorded decision.
6. Decide: accept, reject with specific feedback, or request clarification.

The user's decision is final. If the user rejects the output, the design lead must re-dispatch the task with updated instructions addressing the user's feedback.

## 9. Task Return Conditions

A task must be returned (not accepted) under any of these conditions:

1. The agent modified a `forbidden file`.
2. The agent skipped a `required reads` file and the output shows ignorance of its content.
3. The agent dropped a blocked item from a prior gate without resolution.
4. The agent weakened a confirmed requirement without a `DECISIONS.md` entry.
5. The agent claimed a gate has passed (gate passage is determined by the gate model, not by the agent).
6. The agent declared V1.1 complete or started work on V1.2+.
7. The output artifact does not exist at the specified `evidence output path`.
8. The final report does not follow the required format.
9. The agent exceeded its `allowed files` boundary.
10. The agent's output contains fabricated status claims (e.g., "accepted", "verified", "closed") that were not determined by the gate process.

When a task is returned, the design lead must:

1. Record the return reason in the gate record for the relevant stage.
2. Update the dispatch with specific correction instructions.
3. Re-dispatch to the same or a different agent.

A returned task does not count as a completed gate. The stage remains blocked until a valid output is accepted.
