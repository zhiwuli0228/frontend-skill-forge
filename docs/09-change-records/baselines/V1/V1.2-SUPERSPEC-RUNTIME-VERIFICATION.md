## Verified Runtime Behaviors

The following behaviors are now evidenced:

1. OpenSpec recognizes `superspec` as a valid project-local custom schema.
2. OpenSpec binds a new change shell to `superspec` through `.openspec.yaml`.
3. OpenSpec computes the declared artifact dependency chain:
   `brainstorm -> proposal -> specs -> design -> design-review -> design-disposition -> design-closure -> plan -> tasks -> implementation -> implementation-review -> implementation-disposition -> implementation-closure -> verify -> verify-review -> verify-disposition -> verify-closure -> acceptance-precheck -> release`
4. OpenSpec resolves all declared templates from the repository-local schema directory.
5. OpenSpec emits artifact instructions that include template content, output path, dependencies, and unlocked successors.
6. The `verify` artifact is runtime-recognized as a first-class stage rather than an implementation note.

## Explicit Non-Claims

This verification does not yet prove:

- that every generated artifact content is semantically sufficient
- that a full human/AI execution run has been completed end to end in the main repository
- that archive/apply behavior matches every repository governance expectation

## Conclusion

After the 2026-06-07 correction, `superspec` is no longer only a documented target shape. It is now validated as a runtime-recognized OpenSpec custom schema with working change binding, status resolution, template resolution, and instruction generation.
