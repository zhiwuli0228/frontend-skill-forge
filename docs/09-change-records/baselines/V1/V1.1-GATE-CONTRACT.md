# V1.1 Gate Contract

Status: Proposed

Version: V1.1-GATE-CONTRACT

Date: 2026-06-07

## 1. Purpose

Define the hard gate model for V1.1 Governance Runtime Foundation. This contract specifies which stage transitions require explicit gate passage, what each gate consumes and produces, and how failures are handled. It is the authoritative reference for determining whether V1.1 work may advance from one stage to the next.

## 2. V1.1 Goal

V1.1 must define and stabilize the governance runtime boundaries for V1. This means producing the documents that later minor versions (V1.2, V1.3, V1.4) will execute against, without claiming the governance model is active.

## 3. V1.1 Boundary

In scope:

- refined V1 IR (this IR already exists; refinement may update status or scope notes)
- refined V1 SR (this SR already exists; refinement may update status or scope notes)
- governance design baseline (already exists as DESIGN-001)
- gate contract (this document)
- dispatch contract (V1.1-DISPATCH-CONTRACT.md)

Out of scope:

- SuperSpec schema implementation (belongs to V1.2)
- Superpower mission and power implementation (belongs to V1.3)
- integration and gate verification walkthrough (belongs to V1.4)
- any `src/`, `tests/`, or `openspec/` code changes
- claiming V1.1 is complete or accepted

## 4. Hard Gates

Each gate below is mandatory. A stage is not complete until its gate has passed. Document existence alone does not satisfy a gate.

### 4.1 IR Review Gate

| Field | Content |
| --- | --- |
| Input | `baselines/V1/IR.md`, `current/DECISIONS.md` |
| Output | IR review record under `baselines/V1/` (e.g. `V1.1-IR-REVIEW.md`) |
| Pass condition | Reviewer confirms: (a) IR covers all user-confirmed requirements in scope, (b) no confirmed requirement is silently missing, (c) out-of-scope items are explicitly listed, (d) IR status is updated to reflect review outcome |
| Failure handling | Record specific gaps in the review record. IR must be revised before re-review. Stage remains blocked. |
| Blocked record | The review record must state "IR review: blocked" with the gap list and the date of blocking. |

### 4.2 IR Disposition Gate

| Field | Content |
| --- | --- |
| Input | IR review record, `current/DECISIONS.md` |
| Output | IR disposition record (may be combined with the review record if clearly separated) |
| Pass condition | Each gap from IR review is classified as: accepted-as-is, resolved, deferred with user confirmation, or blocking. All blocking items are resolved or explicitly user-accepted. |
| Failure handling | If any blocking item lacks resolution or user acceptance, disposition fails. Record the outstanding item and the required decision. Stage remains blocked. |
| Blocked record | The disposition record must state "IR disposition: blocked" with the unresolved item and the decision owner. |

### 4.3 IR Closure Verification Gate

| Field | Content |
| --- | --- |
| Input | IR review record, IR disposition record, updated `IR.md` |
| Output | IR closure verification record |
| Pass condition | Verifier confirms: (a) all disposition actions are reflected in the final IR, (b) no gap was silently dropped, (c) IR status is set to a closed or accepted state, (d) `DECISIONS.md` captures any user-confirmed exceptions |
| Failure handling | Record the discrepancy. IR must be corrected before re-verification. Stage remains blocked. |
| Blocked record | The closure record must state "IR closure: blocked" with the discrepancy and required correction. |

## 5. Blocked State Recording

All blocked states follow the same convention:

1. The gate output document must include the exact phrase `<gate name>: blocked` in its status field.
2. The document must list: the blocking item, the reason it blocks, the date of blocking, and the decision owner (user or design lead).
3. A blocked gate blocks all downstream gates. No later gate may claim passage while an earlier gate is blocked, unless `DECISIONS.md` contains an explicit user decision granting a recorded exception.
4. Blocked items must be carried forward into every subsequent gate record until resolved. They may not be silently dropped.

## 6. Minor Version Exit Condition

V1.1 may be considered complete (ready for user acceptance review) when:

1. All 11 gates listed above have passed with non-blocked records.
2. All V1.1 required outputs exist under `baselines/V1/`:
   - `V1.1-GATE-CONTRACT.md` (this document)
   - `V1.1-DISPATCH-CONTRACT.md`
   - `V1.1-IR-REVIEW.md` (or equivalent gate record)
   - `V1.1-IR-DISPOSITION.md`
   - `V1.1-IR-CLOSURE.md`
   - `V1.1-SR-REVIEW.md`
   - `V1.1-SR-DISPOSITION.md`
   - `V1.1-SR-CLOSURE.md`
   - `V1.1-IMPLEMENTATION-REVIEW.md`
   - `V1.1-IMPLEMENTATION-DISPOSITION.md`
   - `V1.1-IMPLEMENTATION-CLOSURE.md`
   - `V1.1-ACCEPTANCE-PRECHECK.md`
   - `V1.1-ACCEPTANCE-PRECHECK-VERIFICATION.md`
3. `DECISIONS.md` reflects all user-confirmed exceptions.
4. No gate record shows "blocked" status.

User acceptance of V1.1 is the final gate. The design lead may not declare V1.1 accepted; only the user may do so.

## 7. Major Version Progression Rule

V1.2 may not start until the user has explicitly accepted V1.1.

**No user acceptance, no next minor version.**

**No user acceptance, no major version closure.**

This rule is absolute and has no automation bypass. Passing all gates is necessary but not sufficient. The user's explicit acceptance is the final gate for both minor-version completion and major-version progression.
