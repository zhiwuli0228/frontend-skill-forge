# V1.2 SuperSpec Activation Contract

Status: Closed for V1.2

Version: V1.2-CONTRACT

Date: 2026-06-07

## Purpose

Define what must be true before this repository may claim that `superspec` is active.

## Activation Preconditions

The repository may claim `superspec` activation only when all of the following exist:

1. `openspec/schemas/superspec/schema.yaml`
2. runnable templates for each required stage artifact
3. a workflow reference that matches the V1 gate model
4. usage guidance aligned to the implemented schema
5. validation evidence showing the local schema is recognized by the OpenSpec toolchain

## Non-Activation Rule

Any of the following states means activation is not yet allowed:

- only `openspec/config.yaml` names `superspec`
- design docs exist but no schema files exist
- templates exist but do not cover review/disposition/closure flow
- activation claims exist without validation evidence

## Required Evidence for Future Activation

When code changes become allowed, activation evidence must include:

- created file list under `openspec/schemas/superspec/`
- schema validation command output
- example change artifact generation record
- mismatch log if any stage is not yet covered

## Current Validation Boundary

The V1.2 correction for directory reliability is:

- `openspec schema which superspec` must resolve to `openspec/schemas/superspec/`
- `openspec schema validate superspec` must pass against the current schema file
- companion files such as `README.md` and `references/workflow.md` are explanatory only and are not evidence of OpenSpec compatibility

Additional runtime evidence captured on 2026-06-07 shows that:

- `openspec new change ... --schema superspec` succeeds in an isolated verification workspace
- `openspec status --change <id> --json` resolves the declared gate dependency chain
- `openspec templates --schema superspec --json` and `openspec instructions <artifact> ...` resolve project-local templates correctly

See [V1.2-SUPERSPEC-RUNTIME-VERIFICATION.md](./V1.2-SUPERSPEC-RUNTIME-VERIFICATION.md).

## V1.2 Conclusion

V1.2 does not activate `superspec`. It defines the contract that future activation must satisfy.
