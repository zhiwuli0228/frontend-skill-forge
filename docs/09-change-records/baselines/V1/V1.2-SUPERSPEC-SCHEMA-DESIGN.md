# V1.2 SuperSpec Schema Design Baseline

Status: Closed for V1.2

Version: V1.2-DESIGN

Date: 2026-06-07

## Purpose

Define the repository-local `superspec` schema structure, stage semantics, and artifact responsibilities, including the static schema assets stored under `openspec/schemas/superspec/`.

## V1.2 Goal

Produce an activation-ready SuperSpec baseline, including static schema files and templates, so that later runtime verification can be done without reopening V1 governance questions.

## Design Boundary

In scope:

- schema directory blueprint
- static schema files
- static templates
- workflow reference
- stage model
- artifact map
- activation prerequisites
- usage guidance

Out of scope:

- business/frontend implementation
- end-to-end lifecycle automation verification
- changing OpenSpec CLI behavior

## Stage Semantics

| Stage | Meaning | Required Control |
| --- | --- | --- |
| brainstorm | Explore scope and constraints before change creation | Superpower-owned skill input |
| proposal | Define scope, constraints, and required evidence | Must name verification intent |
| specs | Define functional and browser-observable requirements | Must map to E2E scenarios |
| design | Define frontend implementation and verification design | Must include skill and MCP usage |
| review/disposition/closure | Independent gate sequence | Hard gate, no bypass |
| plan | Define execution ordering and evidence checkpoints | Must identify skill and tool usage |
| tasks | Define trackable work units | Must pair implementation with E2E verification |
| implementation | Record bounded execution | Must hand off to dedicated verify stage |
| verify | Gather executed evidence | No silent downgrade; no unit-only acceptance |
| acceptance precheck | Reverse traceability | Required before release claim |
| release | Major/minor summary | User acceptance required for major closure |

## Skill And Path Contract

The schema contract also fixes:

1. Which skill family is allowed to generate each artifact.
2. Which relative output path each artifact must use inside `openspec/changes/<change>/`.
3. Which artifacts are forbidden from being redirected into `docs/` or other repository-local narrative locations.

Examples:

- `brainstorm` remains at `brainstorm.md`, but that file may serve as a redirect manifest to the canonical Superpowers output under `docs/superpowers/specs/`.
- `proposal`, `design`, `plan`, `tasks`, `implementation`, `verify`, and gate artifacts each have one fixed filename under the change root.
- `specs` may contain multiple files, but all of them must remain under `specs/`.

## Design Rules

1. `superspec` must reference the repository gate model rather than a lightweight default OpenSpec flow.
2. The schema must preserve separation of concerns: OpenSpec for change lifecycle, SuperSpec for workflow, Superpower for brainstorm and dispatch.
3. The schema must be able to point into `docs/09-change-records/` for authoritative baseline outputs.
4. The schema file itself must stay within the official OpenSpec custom-schema contract so `openspec schema validate` can verify it.
5. Companion documents may extend explanation, but not the parser-visible structure that OpenSpec validates.
6. User-visible frontend changes must require verification through the repository's existing Playwright E2E framework, not only lint, build, or unit tests.
7. Major execution artifacts must record required skills and MCP/tool usage when those are part of the workflow.
8. Artifact generation skills and output paths must be deterministic to prevent process drift across agents.
9. The schema must not imply full activation until runtime validation evidence exists.
