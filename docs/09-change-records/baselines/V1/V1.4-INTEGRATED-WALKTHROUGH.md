## Purpose

Record the integrated dry run showing how OpenSpec, SuperSpec, and Superpower now fit together under the V1 gate model.

## Walkthrough Scope

This walkthrough verifies the control plane, not business/runtime feature delivery. It checks that:

1. OpenSpec sees the repository-local `superspec` schema.
2. SuperSpec exposes the expected workflow templates and artifact map.
3. Superpower provides the governing rules for brainstorm, dispatch, review, and closure.
4. The repository still builds and the current smoke E2E remains intact after the governance-layer additions.

## Integrated Flow Summary

### Step 1: Change engine binding

`openspec project.md` and the validated `superspec` schema establish that OpenSpec remains the entrypoint while the repository-local schema controls workflow shape.

### Step 2: Workflow-stage resolution

Template resolution proves the declared stages can be mapped to concrete project-local artifacts, including review, disposition, closure, and acceptance-precheck templates.

### Step 3: Agent-governance overlay

The Superpower mission, power, brainstorm, and dispatch documents provide the operator rules that explain how an agent should behave before, during, and after formal stage work.

### Step 4: Repository stability check

Build and smoke-test results show that adding the governance control plane did not regress the current runtime baseline.

## Residual Limits

This walkthrough does not prove:

- that a full nontrivial change was executed end to end in this repository during V1.4
- that business/frontend implementation is now allowed
- that every future orchestrator honors custom skill routing perfectly

## Conclusion

At V1.4 scope, the repository now has a coherent control-plane integration: OpenSpec provides change entry, SuperSpec provides workflow contract, and Superpower provides governed agent behavior. The remaining gap is major-version acceptance, not control-plane definition.
