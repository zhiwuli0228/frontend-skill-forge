## 3. Hotfix: Sidebar Filter Navigation (post-V2.10)

After the modules were complete, the user reported that sidebar items had "no effect" when clicked. Root cause: 4 of 5 modules' sidebar items all pointed to the same URL, so `navigate()` succeeded but the URL didn't change.

### Fix

- Router: `path: 'list'` → `path: 'list/:filter?'` (optional param) for 4 modules
- `moduleConfig.tsx`: each sidebar item gets a unique path
- Workflow module: sidebar changed from node types (trigger/process/condition/output) to status filters (active/inactive/draft) so they map to real data
- `SidebarNavigation.tsx`: `selectedKeys` derived from current pathname (was hard-coded `[]`)
- Pages: read `useParams().filter` and filter mock data; show a `sidebar-filter-indicator` tag
- New `archived` status added to task data with 2 archived tasks
- Insight overview: time-range param scales stat-card values and chart data (today=5%, week=25%, month=100%, quarter=300%, all=600%)

### Verification

- 13 new e2e tests in `sidebar-filter-navigation.spec.ts` covering all 5 modules
- All 181 e2e tests pass
- `npm run lint` and `npm run build` clean

## 4. Pattern Library Established

Across V2.6–V2.10 we standardized these reusable patterns:

| Pattern | Where | Why |
| --- | --- | --- |
| **Scenario selector** | All 18 new pages | Same loaded/loading/empty/error switcher; gives deterministic states for automation |
| **`data-testid` everywhere** | All 18 new pages | Decouples automation from CSS/text; one of the strictest selector strategies |
| **Mock data in `data/` folder** | All 5 modules | No API calls; pages can be exercised in isolation |
| **SidebarMenuItem with `path`** | All 5 modules | Separate navigation key from URL path; solved the duplicate-key warning |
| **Table component as wrapper** | task/workflow/insight | Centralizes empty/loading/error rendering for tables |
| **CSS-only bar charts** | task-templates / insight-overview | Avoids Recharts dependency bloat; still renders meaningful visualizations |
