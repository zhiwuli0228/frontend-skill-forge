# SuperSpec Workflow Reference

## Purpose

Provide the canonical workflow semantics for the repository-local `superspec` schema.

## Stage Flow

```text
brainstorm
-> proposal
-> specs
-> design
-> design review
-> design disposition
-> design closure
-> plan
-> tasks
-> implementation
-> implementation review
-> implementation disposition
-> implementation closure
-> verify
-> verify review
-> verify disposition
-> verify closure
-> acceptance precheck
-> release
```

## Gate Rules

1. A later stage may not claim completion if an earlier required gate is blocked.
2. Review, disposition, and closure must be recorded separately unless the repository explicitly allows a merged record.
3. User-confirmed important requirements must stay traceable through proposal, specs, design, implementation, verify, and acceptance-precheck outputs.
4. User-visible frontend changes must not pass verification using unit-only evidence. Playwright E2E results are mandatory unless a recorded blocker says otherwise.
5. Each artifact must be generated by its schema-assigned skill family. Substituting a different skill without an explicit repository decision is not allowed.
6. Artifact output paths are fixed by `schema.yaml`. Redirection to alternate folders, docs trees, or ad hoc filenames is forbidden unless the schema contract itself is revised.
7. The only standard path-flexible artifact root is `specs/**/*.md`, and every generated spec file must still remain under `openspec/changes/<change>/specs/`.
8. `brainstorm` is the only redirect-managed artifact. Its change-local file stays `openspec/changes/<change>/brainstorm.md`, but when Superpowers generated the brainstorm, that file must redirect only to `docs/superpowers/specs/`.
9. Static schema assets define workflow shape only. They do not prove end-to-end operational validation.
10. If the user explicitly asks for a formal change/spec flow that stops before coding, the repository must complete artifacts only through the `apply` entrypoint and must not substitute version-ledger design docs for change-local artifacts.

## Brainstorm Redirect Audit

The `brainstorm.md` redirect manifest is compliant only when it records all of the following:

- `source_mode`
- `required_skill`
- `canonical_brainstorm_path`
- `redirect_approved`
- `path_compliance`

When `source_mode=fallback-local`, it must also record `fallback_reason`.

## Ownership Rules

- OpenSpec owns change lifecycle entry and artifact generation.
- SuperSpec owns workflow structure and artifact template shape.
- Superpower owns brainstorm behavior, dispatch rules, and execution governance.
- The version ledger under `docs/09-change-records/` remains the authority for current status and release progression.

## Skill Routing Rules

- `brainstorm` should use the Superpowers brainstorming skill when that skill is installed.
- If Superpowers is unavailable, `brainstorm` may fall back to `openspec-explore`, but the fallback and reason must be recorded in `brainstorm.md`.
- `proposal`, `specs`, and `design` must use OpenSpec authoring skills aligned to their artifact type.
- `plan` and `tasks` must use planning/decomposition skills, not direct implementation skills.
- `implementation` must use the approved coding skill for this repository.
- `verify` must use a verification-capable skill that can run Playwright E2E and record evidence.
- `review`, `disposition`, and `closure` artifacts must use their corresponding gate skills and must not be silently merged into implementation or verify outputs.
- If the requested outcome is "handoff at apply", the producing agent must stop once `implementation` becomes the next ready artifact and provide the change name plus context files to the next agent.
